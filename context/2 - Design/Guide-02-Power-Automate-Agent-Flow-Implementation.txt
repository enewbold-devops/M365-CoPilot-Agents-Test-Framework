GUIDE 02: POWER AUTOMATE AND AGENT FLOW IMPLEMENTATION
Microsoft 365 Copilot Studio Agent Framework
Monitoring, Evaluation, and Testing Platform
================================================================================
Date: February 24, 2026
Version: 2.0  (Revised: Dual-Flow Architecture — Adapter + Core split)
Audience: Power Platform Maker, Flow Developer, Integration Lead
================================================================================


PURPOSE OF THIS GUIDE
----------------------
This guide provides the complete step-by-step implementation for the Copilot Studio
topic design and the two cooperating Power Automate flows that process file
attachments submitted through a conversation.

Previous versions of this guide defined a single monolithic Agent Flow (PVA trigger
type) that contained all business logic. That approach was replaced because:
  - PVA-trigger flows can only be invoked from within a Copilot Studio topic
    conversation. They cannot be called directly by the test harness, Azure DevOps
    pipelines, or any external tooling without going through Direct Line first.
  - This forced the test harness (Guide 06) to rely on fragile Direct Line polling
    for all test invocations, including purely functional tests with no conversational
    requirement.
  - Scoring logic existed in two places: inside the flow and inside the harness,
    creating drift risk over time.

The revised architecture introduces a dual-flow split:
  - Flow A: AgentFlow-FileUpload-Adapter  (PVA trigger — called only by the topic)
  - Flow B: CloudFlow-FileUpload-Core     (HTTP trigger — contains all business logic)

The Adapter is a thin wrapper that translates topic variables into the Core flow's
InputContract and relays the OutputContract back to the topic. The Core flow is
callable by anything that can issue an authenticated HTTP POST — Power Automate,
Azure DevOps YAML steps, Postman, or Azure Functions.

This guide covers:
  - How to design the Copilot Studio topic to accept file uploads and pass them
    to the Adapter flow (unchanged from v1.0)
  - The InputContract and OutputContract schemas (new in v2.0)
  - Flow A: AgentFlow-FileUpload-Adapter implementation (new in v2.0)
  - Flow B: CloudFlow-FileUpload-Core implementation (Gates 1 through 7)
  - Gate 7: Scoring, Safety Gate, and OutputContract assembly (new in v2.0)
  - App Insights custom event emission strategy
  - Error handling and retry policy

You should complete Guide 01 (Dataverse Schema and Setup) before implementing
the flows in this guide, because the flows write to the tables defined there.


SECTION 1: COPILOT STUDIO TOPIC DESIGN
---------------------------------------
The topic is the conversational layer. It collects intent and the file from the
user, then hands off to the Agent Flow. The topic itself performs no parsing,
validation, or business logic — that all lives in the flow.

1.1 Topic Name and Trigger
   Name your topic descriptively, for example: "Process File Upload"
   Set the trigger phrases to cover likely user expressions:
     - "I want to upload a file"
     - "upload spreadsheet"
     - "submit my data file"
     - "I have a CSV to import"
     - "process my file"
   You can expand this list over time based on analytics from real sessions.

1.2 Node Sequence in the Topic
   Design the topic with nodes in this order:

   NODE 1: File Question
     - Add a "Question" node using the File attachment input type.
     - Prompt text (example):
       "Please upload your file. I accept CSV, XLSX, and XML formats up to 50 MB."
     - Store the response in a variable: Topic.UploadedFile
     - This variable will contain the file metadata (filename, URL, size, type)
       that Copilot Studio makes available after a user uploads a file.

   NODE 2: Intent Clarification
     - Add a "Question" node with multiple-choice options.
     - Prompt text (example):
       "What would you like to do with this file?"
     - Options:
       * Validate and summarize
       * Import records into the system
       * Validate and return a cleaned file
       * Compare with previous submission
     - Store the response in: Topic.UserIntent
     - This intent is passed to the flow so it can branch logic accordingly.

   NODE 3: Confirmation Message
     - Send a message to the user while the flow executes (this sets expectations):
       "Thank you. I've received your file and I'm processing it now.
        This typically takes 15–60 seconds depending on file size."

   NODE 4: Call Agent Flow
     - Add a "Call an action" node — select your Agent Flow (created in Section 2).
     - Input parameters to pass to the flow:
         ConversationId   = System.Conversation.Id
         SessionId        = System.Conversation.Id  (use same for simplicity)
         UserId           = System.User.Id
         UserDisplayName  = System.User.DisplayName
         AgentId          = (hardcode your bot GUID or use a topic-level variable)
         AgentVersion     = (hardcode your current version string)
         FileUrl          = Topic.UploadedFile.Url
         FileName         = Topic.UploadedFile.Name
         FileSizeBytes    = Topic.UploadedFile.Size
         FileType         = Topic.UploadedFile.MimeType
         UserIntent       = Topic.UserIntent
         IsTestMode       = false   (always false for production topics)
     - Receive these output variables from the flow:
         FlowStatus       (Success / PartialSuccess / Failed)
         TotalRows        (integer)
         ValidRows        (integer)
         InvalidRows      (integer)
         CreatedCount     (integer)
         UpdatedCount     (integer)
         ExceptionReportUrl (text, URL)
         SummaryMessage   (text, formatted for display)
         BatchUploadId    (GUID of the new_BatchUpload record)

   NODE 5: Branching on FlowStatus
     - Add condition branches:
       * If Topic.FlowStatus equals "Success":
           Show success message using SummaryMessage variable.
           Offer next steps (download exception report, route, etc.)
       * If Topic.FlowStatus equals "PartialSuccess":
           Indicate partial success, show counts, link to exception report.
       * If Topic.FlowStatus equals "Failed":
           Explain the failure reason.
           Offer to retry or escalate to a human agent.

   NODE 6: Structured Output Message
     - Always return a structured summary regardless of status. Example format:
       "Processing complete:
        - Total rows: {TotalRows}
        - Valid rows: {ValidRows}
        - Invalid rows: {InvalidRows}
        - Records created: {CreatedCount}
        - Records updated: {UpdatedCount}
        - Exception report: {ExceptionReportUrl}
        Next steps: Review the exception report for rows that could not be imported."

1.3 Topic-Level Variables to Initialize
   Initialize these variables at the top of the topic (before Node 1):
     Topic.UploadedFile    — File type
     Topic.UserIntent      — Text
     Topic.FlowStatus      — Text (default: "")
     Topic.TotalRows       — Number (default: 0)
     Topic.ValidRows       — Number (default: 0)
     Topic.InvalidRows     — Number (default: 0)

1.4 Error Handling in the Topic
   Add an "On Error" branch at the topic level. If the Agent Flow call itself fails
   (not a business logic failure, but a flow invocation failure), the On Error branch
   should:
     - Log the system error message (System.LastErrorMessage) to a Dataverse error
       log or App Insights via a separate lightweight flow.
     - Return a user-facing message: "Something went wrong processing your file.
       Our team has been notified. Please try again or contact support."


SECTION 2: DUAL-FLOW ARCHITECTURE
-----------------------------------
The processing pipeline is split across two flows. Together they form a clean
contract boundary: the topic always talks to the Adapter; everything else (test
harness, Azure DevOps, Postman) talks directly to the Core flow.

  Flow A — AgentFlow-FileUpload-Adapter
    Type:    Instant cloud flow — Power Virtual Agents (Copilot Studio)
    Trigger: Called by the Copilot Studio topic via "Call an action" node
    Role:    Thin translation layer. No business logic.

  Flow B — CloudFlow-FileUpload-Core
    Type:    Standard cloud flow
    Trigger: HTTP Request (When an HTTP request is received)
    Role:    All Gate logic (1–7), scoring, EvalResult writes, App Insights events

Read Section 2A first (contracts), then 2B (Adapter), then 2C onward (Core gates).


SECTION 2A: INPUT CONTRACT AND OUTPUT CONTRACT
-----------------------------------------------
The InputContract is the JSON body sent TO CloudFlow-FileUpload-Core by either the
Adapter or the test harness. The OutputContract is the JSON body returned synchronously
in the HTTP 200 response. Both schemas must remain stable across deployments — any
additions should be backward-compatible (additive only).

Store the canonical schema versions as Environment Variables in Power Automate
(or in a new_AgentFrameworkConfig Dataverse row) so flows can reference the expected
version and reject mismatched callers cleanly.

INPUTCONTRACT — required fields
---------------------------------
  conversationId      String   Copilot Studio conversation GUID (or synthetic for tests)
  sessionId           String   Same as conversationId for simplicity (or Direct Line session)
  userId              String   Entra ID object ID of the submitting user
  userDisplayName     String   Display name of the submitting user
  agentId             String   Copilot Studio bot GUID
  agentVersion        String   Current version string of the agent (e.g. "2.1.0")
  fileUrl             String   URL to the uploaded file (SharePoint, Blob, or Direct Line CDN)
  fileName            String   Original filename including extension
  fileSizeBytes       Integer  File size in bytes
  fileType            String   MIME type (e.g. "text/csv", "application/vnd.ms-excel")
  userIntent          String   Intent label from topic choice node
  isTestMode          Boolean  true = test run (skip real upserts, mark records as test data)
  evalRunId           String   GUID of the parent EvalRun record. Null/empty for production.
  testCaseId          String   GUID of the TestCase record. Null/empty for production.
  turnsToResolution   Integer  Number of conversational turns before file was submitted

INPUTCONTRACT — JSON structure example
-----------------------------------------
  {
    "conversationId":    "a1b2c3d4-...",
    "sessionId":         "a1b2c3d4-...",
    "userId":            "u-99887766-...",
    "userDisplayName":   "Alex Johnson",
    "agentId":           "bot-guid-here",
    "agentVersion":      "2.1.0",
    "fileUrl":           "https://<tenant>.sharepoint.com/...",
    "fileName":          "Q1_claims.csv",
    "fileSizeBytes":     204800,
    "fileType":          "text/csv",
    "userIntent":        "Import records",
    "isTestMode":        false,
    "evalRunId":         null,
    "testCaseId":        null,
    "turnsToResolution": 3
  }

OUTPUTCONTRACT — fields returned in HTTP 200 body
----------------------------------------------------
  flowStatus          String   "Success" | "PartialSuccess" | "Failed"
  totalRows           Integer  Total rows parsed from the file
  validRows           Integer  Rows that passed all validation rules
  invalidRows         Integer  Rows that failed one or more validation rules
  createdCount        Integer  Records created in the target Dataverse table
  updatedCount        Integer  Records updated in the target Dataverse table
  exceptionReportUrl  String   SharePoint URL of the exception report CSV (empty if none)
  batchUploadId       String   GUID of the new_BatchUpload Dataverse record
  summaryMessage      String   Human-readable formatted summary for display in chat
  aiSummary           String   AI-generated summary string (empty if AI step skipped)
  overallScore        Decimal  0–100 weighted agent score (computed in Gate 7)
  isHardFail          Boolean  true if a Safety Gate condition was triggered
  failureReason       String   Machine-readable failure code or empty string
  latencyMs           Integer  Total flow execution time in milliseconds

OUTPUTCONTRACT — JSON structure example
------------------------------------------
  {
    "flowStatus":         "PartialSuccess",
    "totalRows":          17,
    "validRows":          10,
    "invalidRows":        7,
    "createdCount":       10,
    "updatedCount":       0,
    "exceptionReportUrl": "https://<tenant>.sharepoint.com/.../exceptions_2026-02-26.csv",
    "batchUploadId":      "b9c8d7e6-...",
    "summaryMessage":     "Processing complete for 'Q1_claims.csv': ...",
    "aiSummary":          "10 valid records show expected distribution across regions...",
    "overallScore":       74.5,
    "isHardFail":         false,
    "failureReason":      "",
    "latencyMs":          12340
  }

HTTP authentication for CloudFlow-FileUpload-Core
--------------------------------------------------
The HTTP trigger URL generated by Power Automate includes a Shared Access Signature
(SAS) token in the query string by default. For production use:
  Option 1 (recommended): Store the trigger URL in Azure Key Vault. The Adapter
    flow and the test harness both retrieve the URL from Key Vault at runtime.
    The URL secret should be rotated whenever the Core flow is republished, since
    Power Automate regenerates the SAS token on each publish.
  Option 2: Add a custom API key header check as the first step in the Core flow.
    Use a Power Automate Environment Variable to store the expected key value.
    Reject requests where the header value does not match (respond HTTP 401).
  Use managed identity: If calling from Azure DevOps or an Azure Function, prefer
    managed identity authentication over static SAS tokens for better security posture.


SECTION 2B: FLOW A — AGENTFLOW-FILEUPLOAD-ADAPTER
---------------------------------------------------
This flow is the only flow the Copilot Studio topic interacts with. It has no
business logic — its sole purpose is contract translation.

Creation steps:
  1. In Power Automate, create a new flow.
  2. Select type: Instant cloud flow — Power Virtual Agents (Copilot Studio).
  3. Name it: AgentFlow-FileUpload-Adapter
  4. Connect it to your Copilot Studio environment and agent.

Input parameters (received from the topic — same as the NODE 4 list in Section 1.2):
  ConversationId, SessionId, UserId, UserDisplayName, AgentId, AgentVersion,
  FileUrl, FileName, FileSizeBytes, FileType, UserIntent, IsTestMode
  (IsTestMode is always false when called from the production topic)

Flow steps:

  STEP 1: Retrieve Core flow trigger URL from Key Vault
    - Use the Azure Key Vault connector → "Get secret" action.
    - Secret name: "CoreFlow-FileUpload-TriggerUrl"
    - Store the secret value in a variable: CoreFlowUrl

  STEP 2: Build InputContract JSON (Compose action)
    - Name the Compose action: "Build_InputContract"
    - Expression body:
        {
          "conversationId":    @{triggerBody()['ConversationId']},
          "sessionId":         @{triggerBody()['SessionId']},
          "userId":            @{triggerBody()['UserId']},
          "userDisplayName":   @{triggerBody()['UserDisplayName']},
          "agentId":           @{triggerBody()['AgentId']},
          "agentVersion":      @{triggerBody()['AgentVersion']},
          "fileUrl":           @{triggerBody()['FileUrl']},
          "fileName":          @{triggerBody()['FileName']},
          "fileSizeBytes":     @{triggerBody()['FileSizeBytes']},
          "fileType":          @{triggerBody()['FileType']},
          "userIntent":        @{triggerBody()['UserIntent']},
          "isTestMode":        false,
          "evalRunId":         null,
          "testCaseId":        null,
          "turnsToResolution": 1
        }
    Note: turnsToResolution is not tracked by the Adapter in production.
    The topic could pass it as an additional input variable if needed for
    evaluation enrichment. Default to 1 if not tracked.

  STEP 3: POST to CloudFlow-FileUpload-Core (HTTP action)
    - Method: POST
    - URI: @{variables('CoreFlowUrl')}
    - Headers:
        Content-Type: application/json
    - Body: @{outputs('Build_InputContract')}
    - Authentication: None required if using SAS-embedded trigger URL.
      If using managed identity: set Authentication type to Managed Identity,
      Audience to https://management.azure.com/
    - Important: Ensure the HTTP action timeout is set to PT90S (90 seconds)
      to give the Core flow adequate time for large file processing.

  STEP 4: Parse OutputContract JSON (Parse JSON action)
    - Content: @{body('HTTP_POST_to_Core')}
    - Schema: derive from the OutputContract JSON structure in Section 2A.
      Use "Generate from sample" and paste the example OutputContract.

  STEP 5: Return output variables to the topic
    Map each OutputContract field to the corresponding flow output variable:
      FlowStatus        ← flowStatus
      TotalRows         ← totalRows
      ValidRows         ← validRows
      InvalidRows       ← invalidRows
      CreatedCount      ← createdCount
      UpdatedCount      ← updatedCount
      ExceptionReportUrl ← exceptionReportUrl
      BatchUploadId     ← batchUploadId
      SummaryMessage    ← summaryMessage
      AISummary         ← aiSummary

  Note: overallScore, isHardFail, failureReason, and latencyMs are returned
  in the OutputContract but are not required by the topic display nodes.
  Include them as optional output variables so they can be surfaced in topic
  analytics or a future "score feedback" node if needed.

Error handling in the Adapter:
  - Add a "Configure run after" condition after the HTTP action to catch:
      * HTTP action failure (non-200 response from Core flow)
      * Timeout (Core flow did not respond within 90 seconds)
  - On failure: return FlowStatus = "Failed", SummaryMessage = "Processing could
    not be completed. Please try again or contact support."
  - Do NOT attempt retry in the Adapter. The Core flow handles its own internal
    retries (see Section 4). A second Adapter call would create a duplicate
    BatchUpload record.


SECTION 2C: FLOW B — CLOUDFLOW-FILEUPLOAD-CORE
------------------------------------------------
This is a Standard Cloud Flow with an HTTP Request trigger. It contains all
business logic (Gates 1–7), orchestrates Dataverse writes, emits App Insights
events, and returns the OutputContract synchronously.

Creation steps:
  1. In Power Automate, create a new flow.
  2. Select trigger: When an HTTP request is received.
  3. Name it: CloudFlow-FileUpload-Core
  4. In the trigger, set "Request Body JSON Schema" to the InputContract schema
     from Section 2A. Use "Generate from sample" and paste the InputContract example.
  5. Save the flow once to generate the trigger URL. Copy the trigger URL and
     store it in Azure Key Vault as secret name: "CoreFlow-FileUpload-TriggerUrl".

Variable initialization (immediately after the trigger):
  - FlowStartTime     — set to utcNow() expression
  - ValidationPassed  — Boolean, default: true
  - FailureReason     — String, default: ""
  - StorageUrl        — String, default: ""
  - BatchUploadId     — String, default: ""
  - TotalRows         — Integer, default: 0
  - ValidRows         — Integer, default: 0
  - InvalidRows       — Integer, default: 0
  - CreatedCount      — Integer, default: 0
  - UpdatedCount      — Integer, default: 0
  - ExceptionReportUrl — String, default: ""
  - AISummary         — String, default: ""
  - OverallScore      — Float, default: 0
  - IsHardFail        — Boolean, default: false

All Gate steps (2.1 through 2.7) execute within this flow.
Gate inputs reference the InputContract fields using the trigger body expression:
  @{triggerBody()?['fieldName']}

The flow ends with an HTTP Response action returning the OutputContract (see Gate 7).

2.1 GATE 1: File Type and Size Validation
   This is the first thing the flow does. Fail fast if the file is not acceptable.

   Step: Initialize a variable called ValidationPassed (Boolean, default: true)
   Step: Initialize a variable called FailureReason (String, default: "")

   Check 1 — File type:
     Using a Compose or Condition action, extract the file extension from FileName.
     Accept list: .csv, .xlsx, .xml
     If the extension is not in the accept list:
       - Set ValidationPassed = false
       - Set FailureReason = "Unsupported file type. Only CSV, XLSX, and XML are accepted."
       - Skip to the Return Output step (early exit).

   Check 2 — File size:
     If FileSizeBytes > 52428800 (50 MB):
       - Set ValidationPassed = false
       - Set FailureReason = "File exceeds the 50 MB size limit."
       - Skip to the Return Output step.

   Check 3 — Filename sanity:
     Reject filenames with path traversal characters: "..", "/", "\", ":".
     If any are present:
       - Set ValidationPassed = false
       - Set FailureReason = "Invalid filename."
       - Skip to the Return Output step.

   Note on malware scanning: If your organization has a malware scanning API or
   a connector (e.g., Microsoft Defender for Cloud Apps), add a malware scan step
   here before proceeding to Gate 2. This is optional but strongly recommended
   for any environment processing externally sourced files.

2.2 GATE 2: Raw File Storage
   Before parsing, store the raw file in SharePoint/OneDrive. This ensures you
   always have the original regardless of what happens downstream.

   Step: Use SharePoint — "Create file" action
     - Site: your designated agent file storage library
       (e.g., https://<tenant>.sharepoint.com/sites/AgentArtifacts)
     - Folder path: /RawUploads/{Year}/{Month}/{Day}/{ConversationId}/
     - File name: FileName
     - File content: retrieve from the file URL. Use HTTP action with the file URL
       and the Dataverse or SharePoint connector to download the binary content.
   Step: Capture the SharePoint file URL from the "Create file" output.
     Store in variable: StorageUrl

   Step: Create new_BatchUpload record in Dataverse:
     - new_conversationid: ConversationId
     - new_sourcefilename: FileName
     - new_filetype: (map mime type to choice value)
     - new_filesizebytes: FileSizeBytes
     - new_storageurl: StorageUrl
     - new_status: 100000000 (Pending)
     - new_istestdata: IsTestMode
     - Lookup bind to agent: bots(<AgentId>)
   Step: Capture the new BatchUploadId GUID from the Dataverse create response.

2.3 GATE 3: Parse and Normalize
   Parse the file into a structured row collection. The approach differs by file type.

   For CSV:
     - Use the "Parse CSV" action or a custom connector / Microsoft Dataflows pattern.
     - Extract headers from row 1.
     - Validate that all required columns are present in the header row.
     - If required columns are missing: set ValidationPassed = false, FailureReason
       = "Missing required columns: {list}". Skip to Return Output.
     - Iterate over data rows. For each row, create a JSON object mapping column
       name to value.

   For XLSX:
     - Use the Excel connector "List rows present in a table" or "Get rows" action.
     - Specify the sheet name (your canonical schema defines which sheet is expected).
     - If the expected sheet does not exist: set ValidationPassed = false,
       FailureReason = "Expected sheet '{SheetName}' not found in workbook."
     - Iterate over rows.

   For XML:
     - Use a Parse JSON action after reading the XML (convert XML to JSON using
       Power Automate's xml() and json() functions or a custom connector).
     - Validate that the root node and required child nodes are present.
     - If schema is invalid: set ValidationPassed = false.
     - Extract records from the XML node collection.

   After parsing, you have a collection of raw JSON row objects. Store count in
   variable: TotalRows.

2.4 GATE 4: Row-Level Data Quality Validation
   For each parsed row:
     - Check for required field presence (not null, not empty).
     - Check data types (numeric fields contain numbers, dates parse correctly).
     - Check uniqueness of the NaturalKey field within this upload batch.
     - Check any business-specific rules (e.g., Amount must be > 0, Status
       must be in allowed values list).
     - Collect all validation errors for the row into a list.
     - Classify row as Valid or Invalid based on error list.
     - Write a new_BatchRow record to Dataverse for each row:
         new_batchuploadid: BatchUploadId
         new_rownumber: row index
         new_naturalkey: value of natural key field in this row
         new_payloadjson: full row as JSON string
         new_validationstatus: 100000000 (Valid) or 100000001 (Invalid)
         new_validationerrors: comma-separated error descriptions (if any)
       Use the PATCH upsert pattern with the alternate key to safely handle retries.

   After the loop:
     - Count valid rows → ValidRows variable
     - Count invalid rows → InvalidRows variable

2.5 GATE 5: AI Step (Optional — Only After Deterministic Pass)
   This step is only executed if ValidationPassed is true AND ValidRows > 0.
   Never run AI logic before deterministic validation.

   Option A — Summarization:
     Pass a sample of valid rows to a GPT-based AI Builder prompt or Azure OpenAI
     connector. The prompt asks for a summary and anomaly highlights.
     Example prompt: "Review the following records and identify:
       1) Overall summary
       2) Any statistical anomalies
       3) Data quality observations
       Records: {first 20 valid rows as JSON}"

   Option B — Entity extraction (for PDF/DOCX scenarios):
     Pass the extracted text to an AI prompt for key entity extraction.
     Store extracted entities in the BatchUpload record or as a separate output.

   Option C — Classification and routing:
     Pass summary info to a classifier prompt to determine routing destination.
     Return routing recommendation as part of SummaryMessage.

   Store AI output in: AISummary variable (string).

   Note: If the AI step fails (timeout, content filter, API error), log the error
   to App Insights but do not fail the entire flow. The deterministic result is
   still valid and should be returned to the user. The AI layer is enrichment only.

2.6 GATE 6: Upsert Valid Rows and Generate Exception Report
   Step: For each valid row, perform the business action:
     - If UserIntent = "Import records": upsert into target Dataverse table using
       the alternate key pattern described in Guide 01. Record the outcome
       (Created / Updated / Skipped / Failed) in the new_BatchRow record by
       issuing a PATCH to update new_upsertaction.
     - If UserIntent = "Validate and return file": skip upsert. Proceed to
       exception report generation.

   Step: Generate exception report:
     If InvalidRows > 0:
       - Create a CSV string from the invalid rows with an added column "Errors".
       - Use SharePoint "Create file" to write the exception report to:
         /ExceptionReports/{ConversationId}/exceptions_{timestamp}.csv
       - Capture URL → ExceptionReportUrl variable.
     If InvalidRows = 0:
       - ExceptionReportUrl = "" (empty)

   Step: Update new_BatchUpload record:
     PATCH the BatchUploadId record with final status, counts, and exception URL.

2.7 GATE 7: Score, Apply Safety Gate, Write EvalResult, and Return OutputContract
   This is the final step in CloudFlow-FileUpload-Core. It executes after all
   prior gates (including Gate 6) regardless of whether they passed or failed,
   so a complete OutputContract is always returned.

   STEP A — Compute latency:
     - Set LatencyMs = int(sub(ticks(utcNow()), ticks(variables('FlowStartTime')))) / 10000
       (converts ticks difference to milliseconds)

   STEP B — Apply Guide 03 weighted scoring formula:
     The score is computed from gate-level metrics already stored in variables.
     Use the following weights (authoritative weights defined in Guide 03 Section 2):

       TaskSuccess score (30%)
         If FlowStatus = "Success": TaskSuccessScore = 100
         If FlowStatus = "PartialSuccess": TaskSuccessScore = 50
         If FlowStatus = "Failed": TaskSuccessScore = 0

       WorkflowCorrectness score (25%)
         UpsertSuccessRate = if(ValidRows > 0, (CreatedCount + UpdatedCount) / ValidRows, 1)
         WorkflowScore = UpsertSuccessRate * 100

       AnswerQuality score (20%)
         GroundingScore: set to 100 if AISummary is non-empty and AI step ran.
         Set to 0 if AI step was skipped (acceptable — weight is applied proportionally).
         If AI step was skipped entirely, set GroundingScore = null and adjust weight
         to redistribute: split the 20% across TaskSuccess and WorkflowCorrectness.

       UX/Latency score (15%)
         If LatencyMs <= SLO (from AgentTestConfig or Environment Variable):
           LatencyScore = 100
         If LatencyMs > SLO but <= 2x SLO:
           LatencyScore = 50
         If LatencyMs > 2x SLO:
           LatencyScore = 0

       Safety score (10%)
         Default: SafetyScore = 100
         Applied by Safety Gate in Step C below.

     OverallScore = (TaskSuccessScore * 0.30)
                  + (WorkflowScore * 0.25)
                  + (GroundingScore * 0.20)   [or redistributed if AI skipped]
                  + (LatencyScore * 0.15)
                  + (SafetyScore * 0.10)

     Implement via a series of Power Automate "Set variable" actions using
     float() and div() expressions. Store result in OverallScore variable.

   STEP C — Apply Safety Gate override:
     Check each hard-fail condition. If ANY is true:
       - Set IsHardFail = true
       - Set OverallScore = 0
       - Set FlowStatus = "Failed" (override any PartialSuccess)

     Hard-fail conditions (from Guide 03 Section 4):
       HF-01: Cross-user data leakage detected (flag set by a dedicated check step)
       HF-02: Silent truncation of field data (truncation occurred without error entry)
       HF-03: Safety severity >= High (from AI Builder content moderation output)
       HF-04: Prompt injection caused agent behavior change (flag set by topic sensor)
       HF-05: Upsert wrote to wrong tenant/user record (detected via ownership check)

     For each condition, add a Condition action checking the relevant flow variable.
     Set FailureReason to the matching code string (e.g., "CrossUserLeakage").

   STEP D — Write EvalResult to Dataverse (conditional):
     Condition: evalRunId field in InputContract is not null AND not empty.
     If true:
       - Create a new_EvalResult record:
           new_evalrunid:           evalRunId (from InputContract)
           new_testcaseid:          testCaseId (from InputContract)
           new_conversationid:      conversationId
           new_agentid:             agentId
           new_agentversion:        agentVersion
           new_tasksuccess:         (FlowStatus = "Success")
           new_overallscore:        OverallScore
           new_ishardfail:          IsHardFail
           new_failurereason:       FailureReason
           new_latencyms:           LatencyMs
           new_totalrows:           TotalRows
           new_validrows:           ValidRows
           new_invalidrows:         InvalidRows
           new_createdcount:        CreatedCount
           new_updatedcount:        UpdatedCount
           new_exceptionreporturl:  ExceptionReportUrl
           new_batchuploadid:       BatchUploadId
           new_aisummary:           AISummary (truncated to 4000 chars if needed)
       Note: The test harness (Guide 06) reads OverallScore and IsHardFail directly
       from the OutputContract. It does NOT recompute the score. The EvalResult record
       is the durable audit store; the OutputContract is the synchronous response.

   STEP E — Build SummaryMessage:
     Compose the human-readable message for display in chat:
       "Processing complete for '{FileName}':
        - Rows processed: {TotalRows}
        - Valid: {ValidRows} | Invalid: {InvalidRows}
        - Records created: {CreatedCount} | Updated: {UpdatedCount}"
     Append if InvalidRows > 0:
       "- Exception report: {ExceptionReportUrl}"
     Append if IsHardFail = true:
       "- Warning: A compliance issue was detected. Please contact your administrator."

   STEP F — Build and return OutputContract:
     Add an HTTP Response action (Respond to an HTTP request):
       Status Code: 200
       Headers:
         Content-Type: application/json
       Body (Compose expression building the OutputContract JSON):
         {
           "flowStatus":          @{variables('FlowStatus')},
           "totalRows":           @{variables('TotalRows')},
           "validRows":           @{variables('ValidRows')},
           "invalidRows":         @{variables('InvalidRows')},
           "createdCount":        @{variables('CreatedCount')},
           "updatedCount":        @{variables('UpdatedCount')},
           "exceptionReportUrl":  @{variables('ExceptionReportUrl')},
           "batchUploadId":       @{variables('BatchUploadId')},
           "summaryMessage":      @{variables('SummaryMessage')},
           "aiSummary":           @{variables('AISummary')},
           "overallScore":        @{variables('OverallScore')},
           "isHardFail":          @{variables('IsHardFail')},
           "failureReason":       @{variables('FailureReason')},
           "latencyMs":           @{variables('LatencyMs')}
         }
     Important: The HTTP Response action must be the absolute last action in the
     flow. Power Automate will immediately return the response to the caller when
     this action executes, terminating the flow run for the caller. App Insights
     events (Section 2.8) should therefore be emitted BEFORE this response action.

2.8 Emit App Insights Custom Events
   Use the HTTP action to POST to your App Insights instrumentation endpoint:
   Endpoint: https://dc.services.visualstudio.com/v2/track
   This is the direct ingestion URL. Use a flow action or custom connector.
   Alternatively, use an Azure Function as a telemetry relay if you need enrichment.

   Emit the following named events:

   EVENT: UploadReceived
     Properties:
       conversationId, agentId, agentVersion, channel, fileType,
       fileSizeBytes, isTestMode, timestamp

   EVENT: ParseCompleted (or ParseFailed if Gate 3 failed)
     Properties:
       conversationId, fileType, totalRows, parseSuccess (bool), latencyMs,
       failureReason (if failed)

   EVENT: ValidationCompleted
     Properties:
       conversationId, totalRows, validRows, invalidRows, validationPassRate

   EVENT: UpsertCompleted (if import intent)
     Properties:
       conversationId, createdCount, updatedCount, skippedCount, failedCount,
       upsertCorrectness (float: successfulUpserts / totalValidRows)

   EVENT: FlowCompleted
     Properties:
       conversationId, flowStatus, totalLatencyMs, isTestMode, overallScore,
       isHardFail, failureReason
     Note: overallScore is sourced from the OverallScore variable computed in
     Gate 7. It is emitted here before the HTTP Response action fires.

   Critical correlation dimension: Always include ConversationId AND AgentVersion
   in every event. This allows every App Insights query to join to Dataverse
   records by ConversationId and to slice performance trends by agent version.

   Placement: All App Insights HTTP actions must appear BEFORE the HTTP Response
   action in Gate 7 Step F. Once the response fires, the flow terminates the
   synchronous call and any subsequent actions may not execute reliably.


SECTION 3: IsTestMode FLAG PATTERN
------------------------------------
The IsTestMode field in the InputContract controls test-safe execution within
CloudFlow-FileUpload-Core. The behavior is defined inline within each gate:

  Gate 1, 2, 3, 4 — execute identically in test and production mode.
  Gate 5 (AI)     — skip in regression (deterministic-only) runs;
                    execute when specifically testing AI quality.
  Gate 6 (upsert) — when isTestMode = true: skip real writes to business tables.
                    Compute and log the counts but write to a test namespace only.
                    All BatchUpload and BatchRow records get new_istestdata = true.
                    SharePoint files go to /RawUploads/TestRuns/{EvalRunId}/.
  Gate 7 (score)  — always executes. EvalResult is written whenever evalRunId
                    is non-null (test run). App Insights events always emit.

This means the test harness exercises 95% of the production code path, making
results a reliable signal of production behavior without contaminating real data.
See Guide 06 Section 6 for the automated cleanup job that removes test records.


SECTION 4: ERROR HANDLING AND RETRY POLICY
---------------------------------------------
Configure the flow with a robust error handling strategy:

Flow-level: Set "Run after" conditions on each Gate block to handle failures
  gracefully. Use a Scope action to wrap each Gate so failures are catchable.

  Try/Catch pattern using Scopes:
    - Create a Scope called "Gate3-Parse".
    - After the Scope, add a condition: "Did Scope 'Gate3-Parse' fail?"
    - If it failed: log the error to App Insights (FlowCompleted event with
      flowStatus="Failed"), update BatchUpload record status to Failed,
      return FlowStatus = "Failed" to the topic.

Retry policy for Dataverse connector:
  Dataverse actions in Power Automate support built-in retry. Set:
    Retry policy: Exponential interval
    Retry count: 3
    Retry interval: PT5S (5 seconds)
    Maximum interval: PT1M (1 minute)
  This handles transient Dataverse throttling (HTTP 429) automatically.

Timeout for the Core flow:
  CloudFlow-FileUpload-Core is a Standard Cloud Flow with an HTTP trigger. Its
  default synchronous timeout is 120 seconds (2 minutes). For large files:
    - Set a hard flow timeout of 240 seconds at the flow level (configurable via
      flow settings > Run After > Timeout).
    - The Adapter flow's HTTP action timeout (Step 3) must be >= Core flow timeout.
      Set it to PT90S minimum; adjust upward if large file processing is expected.
    - If a file regularly exceeds the timeout, consider an async pattern:
        * Core flow starts processing and immediately returns HTTP 202 Accepted
          with a BatchUploadId.
        * A separate "Check status" topic/flow lets the user poll by BatchUploadId.
        * This trades synchronous simplicity for large-file reliability.

SECTION 6: CONNECTING THE FLOW TO ADDITIONAL SCENARIOS
---------------------------------------------------------
The same Agent Flow can serve multiple attachment scenarios by branching on UserIntent:

Scenario A: Upload PDF → Summarize → Create ticket
  Branch on FileType = "pdf" and UserIntent = "Validate and summarize"
  Skip Gate 3 CSV/XLSX parsing. Instead: extract text from PDF using
  AI Builder Document Processing or a custom connector.
  Run Gate 5 (AI) for entity extraction. Skip Gate 6 upsert, instead create CRM ticket.

Scenario B: Upload RFP → Generate response outline
  Branch on UserIntent = "Generate response"
  Gate 5 uses a specialized prompt for RFP section extraction and compliance matrix.
  Output is a structured outline written to SharePoint.

Scenario C: Upload evidence → Classify and route
  Branch on UserIntent = "Route for review"
  Gate 5 classifies document type and routing destination.
  Gate 6 creates a routing queue item rather than a Dataverse upsert.

Scenario D: Compare against prior upload
  Branch on UserIntent = "Compare with previous submission"
  After Gate 4, retrieve the most recent prior BatchUpload record for the same agent
  and user. Compute diff (added rows, removed rows, changed rows). Return diff summary.
